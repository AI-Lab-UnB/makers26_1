Resumo Técnico — Git & GitHub
Introdução / Objetivo
Git e GitHub são ferramentas fundamentais pra qualquer desenvolvedor que queira organizar, versionar e compartilhar código de forma eficiente. O objetivo desse estudo foi entender como funciona o controle de versão na prática — desde inicializar um repositório localmente até colaborar com outras pessoas usando o GitHub.
Git é um sistema de controle de versão distribuído (VCS). Ele rastreia todas as alterações feitas no código e permite voltar a versões anteriores, criar ramificações paralelas do projeto e trabalhar offline. Cada desenvolvedor tem uma cópia completa do repositório na própria máquina.
GitHub é uma plataforma web que hospeda repositórios Git remotamente. Além de armazenar o código, ela oferece ferramentas de colaboração como issues (rastreamento de bugs e sugestões de features), pull requests e wikis.

Fluxo de Trabalho Local
O ciclo básico de trabalho no Git envolve quatro áreas: o diretório de trabalho, a staging area (área de preparação), o repositório local e o repositório remoto.
O processo segue essa lógica: você modifica arquivos no diretório de trabalho → adiciona as mudanças à staging area → salva essas mudanças no repositório local com um commit → e por fim envia pro remoto.

Resultados — Comandos e Conceitos Técnicos
Configuração inicial
bashgit config --global user.name "Seu Nome"
git config --global user.email "seu@email.com"
Isso configura sua identidade globalmente. Necessário antes de qualquer commit.
Inicializar e rastrear
bashgit init                  # Cria o repositório local (pasta .git oculta)
git status                # Mostra o estado atual dos arquivos
git add nome_arquivo      # Adiciona um arquivo específico à staging area
git add .                 # Adiciona todos os arquivos modificados de uma vez
git commit -m "mensagem"  # Salva as mudanças no repositório local
git log                   # Exibe o histórico de commits (autor, data, hash)
Conectar ao repositório remoto (GitHub)
bashgit remote add origin [URL]   # Conecta o repositório local ao remoto
git push -u origin main       # Envia os commits pro GitHub (primeira vez)
git push                      # Envios subsequentes
git pull                      # Baixa e mescla as mudanças do remoto
git clone [URL]               # Copia um repositório remoto pra máquina local
Ramificação (Branching) e Mesclagem (Merging)
Branches permitem trabalhar em novas funcionalidades ou correções de bug de forma isolada, sem mexer no código principal (geralmente a main).
bashgit branch                    # Lista as branches existentes
git checkout -b nome-branch   # Cria uma nova branch e já muda pra ela
git checkout main             # Volta pra branch principal
git merge nome-branch         # Mescla a branch especificada na atual
No GitHub, esse processo de integrar uma branch à principal é feito através de um Pull Request (PR) — você abre um PR pedindo revisão das suas alterações antes de mesclar.
Segurança e boas práticas

.gitignore: arquivo onde você lista o que o Git deve ignorar — como pastas node_modules/, arquivos .env com chaves de API, arquivos de configuração sensíveis, etc. Isso evita que dados privados sejam enviados pro GitHub acidentalmente.
Autenticação SSH: em vez de digitar usuário e senha toda vez, você gera um par de chaves SSH com ssh-keygen e cadastra a chave pública no GitHub. Mais seguro e prático.
CI/CD: plataformas como a Vercel se integram ao GitHub e fazem deploy automático a cada push — o que se chama de Continuous Deployment. Útil pra projetos web.


Conclusões
O Git resolve um problema clássico no desenvolvimento: como salvar o progresso do código de forma organizada, sem depender de cópias manuais ou nomes de arquivo tipo projeto_final_v3_corrigido.zip. Com ele, cada mudança fica registrada com contexto (quem fez, quando e o porquê via mensagem de commit), e dá pra reverter qualquer coisa se algo quebrar.
O GitHub adiciona a camada colaborativa — múltiplos devs trabalhando no mesmo projeto sem pisar no código um do outro, usando branches e pull requests pra revisar antes de integrar. Fora isso, a integração com ferramentas de deploy como a Vercel torna o fluxo de trabalho bem mais fluido.
No geral, entender Git e GitHub não é opcional na área de tecnologia — é praticamente o mínimo esperado de qualquer desenvolvedor.
