## Git & GitHub: O Básico para Versionamento

O Git é um sistema de controle de versão **distribuído**. Isso significa que cada dev tem uma cópia completa do repositório localmente, o que elimina a dependência total de um servidor central para rastrear o histórico.

### Principais Características
* **Distribuído:** Independência de conexão constante.
* **Branching e Merging:** Fluxos paralelos para implementar novas features sem interferir diretamente em outras pessoas.
* **Colaboração:** Sincronização entre repositórios remotos e locais.
* **Open-Source:** Gratuito e com uma comunidade grande.

### GitHub
O GitHub funciona como o host dos repositórios remotos. Ele disponibiliza uma interface para gerenciar o código e ferramentas que otimizam o trabalho em equipe (como Issues, Actions e a gestão de PRs).

---

## Workflow

Para entender o Git visualizamos o fluxo dos arquivos entre as camadas:

1.  **Working Directory:** Onde o código está sendo editado no dia a dia.
2.  **Staging Area:** Onde os arquivos são preparados e selecionados para o commit.
3.  **Local Repo:** Onde o Git consolida as mudanças e cria o snapshot no histórico local.
4.  **Remote Repo:** Onde o código é enviado para integração com o time.

---

## Comandos Essenciais

* `git init`: Inicializa um novo repositório local.
* `git status`: Verifica o estado dos arquivos (quais foram modificados ou estão em staging).
* `git log`: Exibe o histórico de commits.
* `git clone`: Clona um repositório remoto para a máquina local.
* `git remote add origin`: Vincula o repositório local a um servidor remoto.
* `git pull`: Puxa as atualizações do remoto para o local.
* `git checkout -b`: Cria e já alterna para uma nova branch.
* `git push`: Envia os commits locais para o remoto.

## Processos de Integração e Segurança

### Pull Request (PR)
É o fluxo de solicitar que o código de uma branch seja mesclado à branch principal. É o momento crítico para **code review**, garantindo que erros não subam para o projeto principal e gerem problemas futuros e mais complicados de resolver por estar com o que já está certo.

### Segurança com `.gitignore`
Arquivos sensíveis (chaves de API, senhas, `.env`) ou pastas pesadas **nunca** devem ser commitados. O `.gitignore` serve para listar o que o Git deve ignorar sistematicamente.

### Deploy Contínuo
Integrações como **Vercel + GitHub** permitem automação total: qualquer alteração pushada para a branch principal dispara um deploy automático. Isso elimina processos manuais e garante que o ambiente de produção esteja sempre sincronizado com o código mais recente.

### Reflexão Individual
Acredito que seja uma junção de experiência individual com tudo que, enquanto indivíduo, aprendemos na vida e aplicamos ao conhecimento técnico utilizado em ambientes de trabalho profissional.
