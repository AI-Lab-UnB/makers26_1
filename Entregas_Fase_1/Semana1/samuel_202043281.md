### 1. Conceitos básicos e diferenças entre GIT e GITHUB
- O GIT é um sistema de controle de versões descentralizado e distribuído e permite que cada desenvolvedor possua uma cópia completa do repositório localmente, funcionando como uma "máquina do tempo" do código. A maior vantagem é ser OpenSource e gratuito.
- O GITHUB é uma plataforma web para hospedar os repositórios e para organizar os trabalhos colaborativos. O Git oferece funcionalidades como gestão de tarefas, rastreamento de bugs, solicitações de funcionalidades e documentação.
- Esistem outros sistemas de controle de versão alé do git porém são menos utilziados como o SVN e MERCURIAL.


### 2. Principais características do GIT:
- É Descentralizado e distribuído, permite uma cópia local completa do repositório.
- Possui histórico de versões que é um registro de todas as alterações realizadas ao longo do tempo.
- Colaborativo, permite que múltiplos desenvolvedores tabalhem simultaneamente no mesmo projeto sem interferir no trabalho de outro desenvolvedor.
- Cria e genrecia Branches (ramificações), são linhas de desenvolvimento intependentes para funcionalidades ou correções.
- Permite o MERGE que é combinação de códigos de diferentes ramificações.
- Área de preparação (staging area), que é um local intermediaário onde as alterações são organizadas antes de serem registradas no histórico por meio de commit.

### 3. Fluxo de trabalho (Workflow)

| Estado |  Descrição | Comando |
| :--- | :--- | :--- |
| Working Directory | Diretório local onde os arquivos são editados. | - |
| Staging Area | Área de preparação para selecionar arquivos para o próximo commit. | ``git add`` |
| Local Repository | Banco de dados local armazenado na pasta ``.git`` contendo o histórico completo. | ``git commit`` |
| Remote Repository | Servidor remoto (GitHub, GitLab) para sincronização e backup. | ``git push`` / ``git pull`` |

### 4. Configuração inicial
```bash
# Definir identidade do usuário
git config --global user.name "Seu Nome"
git config --global user.email "seuemail@exemplo.com"

# Definir branch padrão como 'main'
git config --global init.defaultBranch main
```


### 5. Comandos Essenciais no Terminal
1. Iniciar Projeto: git init (cria a pasta oculta .git).

2. Verificar Status: git status (identifica arquivos untracked - U; modified - M; added - A).

3. Adicionar ao Stage: git add . (adiciona todos os arquivos modificados/novos) ou git add nome_do_arquivo.

4. Registrar no Repositório Local: git commit -m "Mensagem descritiva".

-  Atalho com flags: git commit -am "Mensagem" (adiciona e comita arquivos já rastreados em um único passo).

5. Visualizar Histórico: git log.

6. Conectar ao Repositório Remoto:

```bash
git remote add origin <URL_DO_REPOSITORIO>
git branch -M main
```
7. Enviar Alterações para o Remoto: `git push -u origin main`.


8. Receber Alterações do Remoto:

```bash
git pull (busca e mescla alterações do remoto no diretório de trabalho).

git fetch (busca alterações do remoto sem mesclar automaticamente).
```

9. Clonar Repositório Existente: `git clone <URL_DO_REPOSITORIO>`.


### 6. Ramificação e Integração (Branch & Merge)
Objetivo: Isolar o desenvolvimento de funcionalidades ou correções de bugs sem impactar a linha principal de código (main).

Pull Request (PR): Mecanismo utilizado no GitHub/GitLab para solicitar a revisão e mesclagem do código de uma branch para outra (ex: `feature/login -> main`).

Forking: Ação de copiar um repositório de terceiros para a sua própria conta no GitHub, permitindo modificações independentes sem afetar o projeto original diretamente.

### 7. Arquivo .gitignore
Utilizado para especificar arquivos ou pastas que não devem ser versionados pelo Git (ex: arquivos de ambiente `.env`, pastas de dependências `node_modules/`, logs). Basta criar um arquivo chamado `.gitignore` e listar os itens a serem ignorados.

# Reflexão sobre "O que é desenvolvimento profissional?"

O desenvolvimento profissional está vai além da capacidade de escrever um código funcional. Está intimamente ligado ao domínio de ferramentas como o GIT e GITHUB que tem como principal objetivo favorecer a organização, documentação e a colaboração ágil e madura.

Dentre as boas práticas temos o uso de branches e PullRequests, que garantem que o código seja feito de forma organizada e segura, já que o merge é protegido pela aprovação e revisão do código mantendo uma rastreabilidade do codigo para que os desenvolvedores saibam o que foi feito e quando foi feito assim facilita em manutenções e melhorias.
