## 1. Resumo Técnico: Git & GitHub baseado no Crash Course 2025

### Git vs. GitHub
* **Git:** É um Sistema de Controle de Versão (VCS) distribuído que funciona localmente na máquina do desenvolvedor, que rastreia mudanças no código, permite reverter alterações e facilita o trabalho em diferentes funcionalidades sem quebrar o projeto principal.
* **GitHub:** É uma plataforma web de hospedagem de repositórios remotos. Enquanto o Git faz o gerenciamento do código, o GitHub facilita a colaboração entre equipes, permitindo Pull Requests, revisão de código (Code Review), gerenciamento de issues e integração contínua.

### O Fluxo de Trabalho (Workflow)
O versionamento profissional segue uma trilha específica de estados de arquivo:
1. **Working Directory:** Onde os arquivos são editados e modificados localmente.
2. **Staging Area:** A "sala de espera". Arquivos prontos para serem salvos são movidos para cá usando o comando `git add`.
3. **Local Repository:** Onde o código é oficialmente registrado no histórico da máquina local através do `git commit`.
4. **Remote Repository:** O repositório na nuvem (GitHub), sincronizado através dos comandos `git push` e `git pull`.

### Principais Comandos Abordados
* `git init`: Inicializa um novo repositório local (cria a pasta oculta `.git`).
* `git clone`: Baixa um repositório remoto para a máquina local.
* `git status`: Verifica o estado atual dos arquivos (rastreados, não rastreados, modificados).
* `git add <arquivo>` ou `git add .`: Move as alterações para a Staging Area.
* `git commit -m "mensagem"`: Salva o estado dos arquivos no repositório local com uma mensagem descritiva.
* `git push`: Envia os commits locais para o repositório remoto.
* `git pull`: Puxa as atualizações mais recentes do repositório remoto para a máquina local.

### Branching e Pull Requests
As **Branches** (ramificações) são cruciais para o trabalho em equipe. Elas criam ambientes isolados (ex: `feature/login`) onde o desenvolvedor pode programar livremente sem afetar a linha principal (`main`). Quando a funcionalidade está pronta, o código é enviado para o GitHub e abre-se um **Pull Request (PR)**, solicitando que a equipe revise o código antes de fundi-lo (merge) com a versão oficial do projeto.

## 2. Reflexão Individual: O que é desenvolvimento profissional?

No contexto do desenvolvimento de software e da cultura proposta pelo AILAB Makers, desenvolvimento profissional vai muito além da simples capacidade de escrever linhas de código. 

Desenvolvimento profissional é a transição da mentalidade de "fazer o código rodar" para a mentalidade de "construir sistemas escaláveis, sustentáveis e colaborativos". Isso envolve disciplina técnica, o uso rigoroso de ferramentas de versionamento profissional (como o Git), e a compreensão de que o código é lido e mantido por outras pessoas.
